Generate a backend coding rules file for **{{FRAMEWORK}}** that covers:

1. **Naming conventions** — variables, functions (verb-first), classes (PascalCase), route paths (plural nouns, hyphen-separated), constants (UPPER_SNAKE_CASE), boolean names (isX, hasX, canX)

2. **Error handling patterns** — global error handler setup, structured error response shape `{ error: { code, message, fields } }`, operational vs programmer errors, retry logic with exponential backoff, circuit breaker pattern for external calls

3. **Security best practices** — input validation (allowlist schema, never trust raw input), authentication middleware placement, authorization checks (resource ownership before returning data), environment variable usage (never hardcode, validate presence at startup), secrets never in logs or error responses

4. **File and folder structure** — group by feature/domain not by technical type, where services, repositories, middleware, and tests live; file naming conventions; test file co-location rules

5. **Import ordering and cleanup** — group order (stdlib → third-party → internal), remove unused imports, aliasing conventions, no circular dependencies between modules

6. **Comments and documentation** — WHY-only comments, no inline what/how comments, docblocks only on public methods and exported functions, TODO/FIXME must include owner and ticket reference

7. **API design conventions** — REST resource naming (plural nouns), HTTP method semantics (GET idempotent, POST creates, PUT replaces, PATCH updates), correct status codes (201 on create with Location header, 204 on delete, 422 for validation, 429 for rate limit), consistent JSON response shape with `data` and `error` wrappers, ISO 8601 UTC for all timestamps

8. **Environment variable usage** — no secrets in source code ever, validate all required env vars at startup and fail fast if missing, never call env() outside of config files, use a secrets manager for production credentials, never log env var values

9. **Logging standards** — structured JSON logs only (no concatenated strings), include request_id in every log line, log levels: error for 5xx, warn for 4xx, info for significant state changes, debug for development only; never log passwords, tokens, PII, or secrets; always include timestamp, level, request_id, and context fields

10. **Database and query hygiene** — always use parameterized queries or an ORM query builder (never string-concatenate SQL), eager-load relationships for list endpoints to prevent N+1, paginate all list queries (no unbounded result sets), select only required columns (never SELECT *), wrap multi-step operations in transactions, set query timeouts

- Output as a single Markdown file suitable for `.cursorrules`, `.mdc`, or `SKILL.md`
- Use `##` for each section heading
- Use bullet points for all rules — no prose paragraphs
- Be specific and opinionated — no "consider doing X", say "always do X" or "never do Y"
- Include short **BAD / GOOD** code examples for non-obvious rules
- Keep each rule to one line where possible — conciseness beats completeness
- No fluff, no introductions, no conclusions — rules only
- Do not include any framework-specific folder names or tool names unless they are idiomatic to {{FRAMEWORK}}
